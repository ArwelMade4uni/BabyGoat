Cross site Scripting


A very common website vulnerability, cross site scripting (XSS), allows an attacker to run malicious code by using a user's connection to a website. This is done when an attacker inputs javascript into an input box or anything where the server will take in information from a user. Attackers can use this to make links to websites with their malicious code embedded inside, so that when someone clicks on that link the code will be run along with the website it's linked with. This is hard to notice unless the malicious code has been set up to pop up or precautions are taken on websites. When these embedded links are clicked, it will be sent to the web server where the website is hosted and run the new code typed by the attacker into the server itself. This is called reflective cross site scripting where the malicious code is reflected back at the user so the code can be executed on the user's device. There's another version of XSS called stored cross site scripting, this is when malicious code is uploaded to the website's server and is stored to be executed every time a user accesses the website. DOM-based XSS is when data is sent from the DOM to a users system and the information sent contains malicious code and is executed on the users system.

Prevention Methods

A solid method into preventing these kinds of attacks is to encode any and all data the user tries to input so that any potential commands are not read as active content. However, this is a timely manor and may require multiple different languages to fully setup. There is a command that can be used to save all uesr inputted content as a selected datatype, this command is 'Content-Type' and allows the developer to change what the users input is saved and read as which will make a normal XSS code read as just a line of text.
